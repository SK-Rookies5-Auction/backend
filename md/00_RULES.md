# 🚀 Secure Auction 프로젝트 협업 규칙

## 1. 브랜치 전략 
- **main**: 최종 배포용 브랜치
- **develop**: 개발 통합 브랜치 (모든 기능이 합쳐지는 곳)
- **feature/기능명**: 각자 기능을 개발하는 브랜치
    - 예: `feature/login`, `feature/bid-logic`

## 2. 커밋 메시지 규칙
커밋 시 반드시 아래 말머리를 달아주세요.
- `feat:` 새로운 기능 추가
- `fix:` 버그 수정
- `docs:` 문서 수정 (README, MD 등)
- `style:` 코드 포맷 변경, 세미콜론 누락 등 (로직 변경 없음)
- `refactor:` 코드 리팩토링
- `chore:` 빌드 업무 수정, 패키지 매니저 설정 등

**예시:** `feat: 로그인 JWT 발급 로직 구현`

## 3. Pull Request (PR) 규칙
- 새로운 기능 완료 시 `develop` 브랜치로 PR을 날립니다.
- 최소 1명 이상의 팀원에게 **Approve(승인)**를 받은 후 직접 Merge 합니다.
- 충돌 발생 시 본인이 해결한 후 올립니다.

## 4. 코드 리뷰 권장 사항
- 로직의 의도가 불분명할 경우 질문을 남깁니다.
